### DEVELOPMENT_GUIDE — 旅游推荐系统 统一开发规范与环境指南
### 项目概述
    - 项目名称：旅游推荐系统 (Tourism Recommendation System)
    - 目标：基于用户的多维度偏好（兴趣、地理位置、出行时间），结合天气、城市热度等动态信息，为用户生成个性化的旅游城市推荐列表。
    - 主要功能模块:
        用户偏好收集：提供界面让用户输入出发地、旅游兴趣标签和计划出行日期。
        推荐引擎：核心算法模块，根据用户偏好、距离、天气、热度等因子为城市综合评分。
        数据服务：提供标准化的 RESTful API，供前端调用。
        数据处理：离线处理和丰富城市数据，为推荐系统提供数据基础。
        性能优化：缓存外部 API 结果（如天气）和常用数据库查询，确保 API 响应速度。
### 架构设计
    - 整体：前后端分离，包含离线数据处理模块的全栈应用。
    - 后端 (backend/):
        技术栈: Spring Boot
        持久层: Spring Data JPA (Hibernate ORM)
        缓存: Redis（兼容使用 Valkey）
        日志: Logback 或 SLF4J
        安全: Spring Security + JWT (身份验证与授权)
        API: RESTful API
    - 包结构建议:
src/main/java
├── com
└── textoasis
├── controller/ # 控制器，处理 HTTP 请求
├── service/ # 业务逻辑层
├── model/ # 数据模型
├── repository/ # 数据访问层，使用 Spring Data JPA
├── config/ # 配置类（例如数据库、缓存、Spring Security 配置）
├── exception/ # 异常处理模块
└── util/ # 工具类
    - 前端 (frontend/):
        技术栈: Vue 3 + Vue Router + Pinia (或 Vuex)
        UI: Bootstrap 5 + 自定义 CSS
        HTTP: Axios (集中在 src/api/ 目录)
        目录结构: src/views, src/components, src/api, src/store, src/router
    - 数据处理 (spark-job/):
        技术栈: Scala + Spark + Maven
        职责：负责计算密集的离线任务，如基于历史数据的游客热度预测模型训练。
    - 数据库:
        使用 MariaDB (兼容 MySQL)。
        部署:
        各服务（后端，前端）均通过 Docker 容器化部署。
        使用 Docker Compose 进行本地一键编排。
        生产环境使用 Kubernetes (K8s) 集群管理。

### 开发规范
- 总体规则:
    代码风格、一致性、注释与接口契约优先于实现细节。
    重要逻辑必须写类 / 方法注释，说明输入、输出、边界条件。
    禁止在版本库提交敏感信息（配置文件中不要包含明文密码 / 密钥）。
- 后端 (Java/Spring Boot):
    命名：类名使用 PascalCase（例如 CityController），变量 / 函数 / 包名使用 snake_case（例如 get_city_data），常量使用 ALL_CAPS（例如 MAX_RETRY_COUNT）。
    接口设计：使用 Spring MVC 或 Spring WebFlux 进行 API 路由，返回统一的 JSON 结构，例如 {"status": "success", "data": ...}。
- 数据库访问：使用 Spring Data JPA 进行 ORM 映射，禁止手写 SQL 查询。
- 日志：使用 SLF4J 或 Logback 进行日志记录，配置好日志级别（如 DEBUG, INFO, WARN, ERROR）。
- 异常：使用自定义异常，并通过 @ControllerAdvice 定义全局异常处理器，返回标准错误格式。
- 前端 (Vue / JS):
    命名：组件用 PascalCase (例如 CityCard.vue)，变量 / 函数用 camelCase（例如 fetchCityData）。
    组件设计：保持单一职责原则，样式使用 scoped 防止全局污染。
    数据与 API: API 调用集中在 src/api/ 目录，基础 URL 等配置使用 .env 文件管理。
- 数据库设计 (MariaDB):
    命名：表名与字段名全小写，下划线分隔 (例如 user_info, city_features)。
    必备字段：每张表必须包含 id (自增主键), create_time, update_time。
    索引：根据查询场景（如按省份查询城市）建立合理索引。
- 日志与异常处理规范
    日志分级:
    DEBUG: 开发调试信息。
    INFO: 核心业务流程，如 “成功生成推荐列表，耗时 X ms”。
    WARN: 潜在风险，如 “天气 API 请求超时，本次评分未计入天气因子”。
    ERROR: 业务或系统异常，如数据库连接失败。
- 日志内容：必须包含时间戳、日志级别、来源模块、消息内容。对于请求处理，建议包含唯一的请求 ID，便于追踪。
- 异常处理:
    后端所有接口异常必须被全局处理器捕获，并返回统一的 JSON 错误响应。
    前端捕获接口异常时，应给用户友好提示（如：“推荐服务开小差了，请稍后重试”），禁止直接暴露技术细节。
- 安全与配置
    配置管理：使用 .env 文件管理本地开发配置，生产环境通过环境变量注入，严禁将生产密码等敏感信息硬编码。
    CORS: 后端配置严格的跨域白名单，仅允许前端域名访问。
    输入校验：所有来自用户的输入都必须经过严格校验，防止注入攻击。
- 性能与可观测
    性能目标:
    推荐接口平均响应时间 < 500ms。
    缓存策略:
    使用 Valkey/Redis 缓存天气 API 的查询结果（同一城市同一天的天气在 1 小时内无需重复请求）。
    缓存常用数据库查询结果，提升接口响应速度。
    可观测:
    集成 Prometheus 监控接口的 QPS 和响应时间，并使用 Grafana 展示。

### Git 管理

此项目使用 Git 进行版本控制。为确保数据的持续更新与追踪，我们将所有城市信息存储在 Git 仓库中，并通过 `GEMINI.md` 文件作为指导，帮助项目开发者更好地使用城市数据。

### 如何使用该数据

1. **导入数据**：将 `GEMINI.md` 文件内容加载到项目中，系统将根据这些城市信息进行推荐。
2. **更新数据**：当有新的城市信息或旅游项目时，只需在 `GEMINI.md` 文件中添加新的记录，然后提交到 Git 仓库。
3. **数据查询**：使用 SQL 或 MongoDB 查询来处理和提取与城市相关的信息，并结合用户的偏好进行个性化推荐。

- 定期备份 Git 仓库，确保数据不会丢失或被误删除。
